Executive Summary
This report presents a comprehensive audit of InsightSwarm v1.0.0, a multi-agent fact-checking system built on LangGraph,
Groq (Llama 3.1 70B), Gemini 2.0 Flash, and Streamlit. The review identified 30 issues spanning architecture logic, source
fetching, LangGraph wiring, token/model handling, testing integrity, and operational reliability. Each issue includes a concrete
fix, and where applicable, a novel enhancement that differentiates InsightSwarm from all existing open-source fact-checkers.
The most critical finding is that source verification runs post-debate and has no effect on the debate outcome — structurally
undermining the system's core value proposition. The recommended path forward centres on three upgrades: retrieval-first
grounding (Tavily RAG), mid-debate verification gates, and heterogeneous-model adversarial pairing.

■ Logical Issues
#01
CRITICAL
Architecture
Verification Runs After Debate — Makes It Purely Cosmetic
■  ISSUE
The FactChecker verifies sources only after all 3 debate rounds complete. Agents never see verification
results and cannot retract hallucinated evidence. The debate outcome is already locked before verification
even starts.
✓  FIX
Insert a VerificationGate node in the LangGraph graph between Round 1 and Round 2. Failed sources are
injected back into agent context as constraints: 'Your source X was not verified. Restate your argument
without it.' Agents are forced to re-anchor to real evidence mid-debate.
★  NOVELTY EDGE
Iterative source-anchored debate is not implemented in any open-source multi-agent fact-checker. This
transforms InsightSwarm from a debate simulator into a genuine evidence-grounded reasoning engine.

#02
CRITICAL
Logic
FactChecker 2x Vote Weight Is Category Confusion
■  ISSUE
FactChecker is a verifier, not a debater. Applying a vote-weight multiplier to a verification report
conflates two fundamentally different agent roles. The '2x weight' is applied to a non-argument, which
corrupts the weighted verdict math.
✓  FIX
Separate the pipeline cleanly: debaters vote (ProAgent, ConAgent), verifier scores (FactChecker).
FactChecker output should be a verification_score (0.0–1.0) that acts as a multiplier on each debater's
argument score, not as a vote itself.
★  NOVELTY EDGE
Treating source verification as an evidence quality multiplier rather than a vote is a cleaner epistemic
model that better reflects how human fact-checkers operate.
#03
CRITICAL
Logic
50% Verified Sources → 'PARTIALLY TRUE' Is Measuring Citation Hygiene, Not Facts
■  ISSUE
The verdict maps source verification rate directly to claim truth value. A perfectly true claim with one
broken URL scores identically to a false claim with one broken URL. The system judges citation quality, not
factual accuracy.
✓  FIX
Decouple verdict from verification rate. Verification rate should flag confidence level, not truth label.
Truth label must come from argument quality assessment. Add a separate confidence_penalty for low
verification rates rather than fusing them into the verdict.
#04
HIGH
Logic
Confidence Score Is Fabricated Precision
■  ISSUE
Output of '68.2%' confidence has no mathematical grounding. LLMs are poorly calibrated on self-reported
confidence. Users making decisions about medical or scientific claims will trust this number as a
probability — it isn't.
✓  FIX
Replace LLM self-reported confidence with a composable score: (arg_quality_score * 0.4) +
(verification_rate * 0.35) + (consensus_agreement * 0.25). Each sub-score is computed deterministically.
Display the breakdown, not just the composite.
★  NOVELTY EDGE
A transparent, decomposed confidence score with visible sub-components is a novel UX differentiator — no
existing consumer fact-checker shows users why it is confident, only how confident it claims to be.

#05
CRITICAL
Architecture
Both Agents Use the Same Model — Adversarial Property Is Fake
■  ISSUE
ProAgent and ConAgent are both Llama 3.1 70B via Groq with different system prompts. They share identical
training data, biases, and knowledge. The Con argument is mathematically the inverse of the Pro argument —
not an independent epistemic perspective.
✓  FIX
Use heterogeneous models: ProAgent on Llama 3.1 70B (Groq), ConAgent on Gemini 2.0 Flash. Different
training data and RLHF alignment produce genuinely divergent arguments. Add a temperature differential: Pro
at 0.3 (factual), Con at 0.7 (exploratory).
★  NOVELTY EDGE
Heterogeneous-model adversarial debate is studied in academic literature but not implemented in any
production open-source fact-checker. This gives InsightSwarm a genuine research novelty claim.
#06
HIGH
Product Logic
No Scientific Consensus Gate — Can Produce 'PARTIALLY TRUE' for Flat Earth
■  ISSUE
The system runs a full adversarial debate for claims with established scientific consensus (vaccines, flat
Earth, climate change). A confident LLM argument can yield 'PARTIALLY TRUE' on a factually settled question
— actively worse than no answer.
✓  FIX
Add a ConsensusPrecheck node before the debate: query WHO, CDC, IPCC, NASA APIs or a curated consensus
knowledge base. If a claim contradicts established consensus with >95% scientific agreement, skip the
debate and return CONSENSUS_SETTLED with authoritative citations.
★  NOVELTY EDGE
A pre-debate consensus gate that routes settled science away from adversarial debate is architecturally
novel and directly addresses a failure mode that has damaged public trust in AI fact-checkers.
#07
HIGH
Logic
No 'Insufficient Evidence' Verdict Path
■  ISSUE
The Moderator always produces a verdict — even when 100% of sources are hallucinated. There is no
abstention path. The system confidently misinforms users on exactly the inputs where it should stay silent.
✓  FIX
Add verdict = INSUFFICIENT_EVIDENCE when: verified_sources == 0 OR total_sources < 2 OR verification_rate <
0.2. Display this as a first-class result with a clear explanation: 'No verifiable sources were found for
this claim. Do not rely on this analysis.'

#08
HIGH
Logic
No Domain Authority — Tabloid = Peer-Reviewed Journal
■  ISSUE
A URL from who.int and one from health-claims-daily.blogspot.com are treated identically if both return
HTTP 200. There is no notion of source credibility, domain authority, or publication type.
✓  FIX
Integrate a DomainTrustScorer: whitelist tier-1 sources (gov, edu, peer-reviewed publishers) with score
1.0; use OpenPageRank API for others; blacklist known misinformation domains. Multiply argument scores by
the average domain trust of their cited sources.
★  NOVELTY EDGE
Domain-trust-weighted argument scoring tied to real authority signals is not present in any open-source
debate-based fact-checker.
#09
HIGH
Logic
Multi-Part Claims Are Silently Mangled
■  ISSUE
A claim like 'Coffee prevents cancer and 5G causes COVID' contains two independent factual assertions.
Agents argue both simultaneously, producing a single verdict that is meaningless for either sub-claim.
✓  FIX
Add a ClaimDecomposer agent as the first graph node. It uses an LLM to split compound claims into atomic
sub-claims, runs a parallel debate subgraph for each, and synthesises a per-claim verdict table. The UI
shows individual verdicts, not one composite.
★  NOVELTY EDGE
Atomic claim decomposition before adversarial debate is a research-grade capability. It mirrors how
professional fact-checkers like PolitiFact handle complex statements.
■ Fetching Issues
#10
CRITICAL
Fetching
BeautifulSoup Blocked by Every Credible Source
■  ISSUE
Nature, Lancet, NEJM, NYT, WaPo all return 403 or empty JS-rendered pages to plain requests.get(). The most
credible sources fail verification. Low-quality sites with no bot-protection pass. The system rewards
low-quality sources.
✓  FIX
Replace raw BeautifulSoup fetching with Tavily Search API or Brave Search API as the primary retrieval
layer. These return clean, pre-extracted content from any domain. Fall back to playwright-based headless
rendering for paywalled URLs, with a 5-second timeout cap.
★  NOVELTY EDGE
Grounding agent arguments in Tavily-retrieved snippets before the debate begins transforms InsightSwarm
from a hallucination-detector to a RAG-powered fact-checker — a fundamental upgrade in information quality.

#11
CRITICAL
Fetching
Agents Invent URLs Post-Hoc to Support Arguments Already Made
■  ISSUE
The flow is: LLM generates argument → LLM invents URLs to support the argument it already made. Citations
are post-hoc rationalisations, not evidence that drove the argument. This is backwards epistemology.
✓  FIX
Flip the pipeline: retrieve first, argue second. Each agent receives a context bundle of 3–5
Tavily-retrieved snippets relevant to their assigned position before generating any argument. Arguments
must be grounded in the provided snippets. The FactChecker then verifies that cited snippets were actually
in the bundle.
★  NOVELTY EDGE
Retrieval-augmented adversarial debate (RAAD) is a novel architecture not implemented in any open-source
system. It eliminates source hallucination structurally rather than trying to detect it after the fact.
#12
HIGH
Fetching
Timeout Labelled as HALLUCINATED — Produces False Positives
■  ISSUE
A connection timeout is labelled identically to a hallucinated URL. Real PubMed studies that time out on
Streamlit Cloud's network get the same HALLUCINATED label as completely invented URLs. This inflates the
hallucination detection rate.
✓  FIX
Implement a 3-state verification result enum: VERIFIED, UNVERIFIED (reachable but content mismatch),
HALLUCINATED (DNS/404 failure), TIMEOUT (network unreachable, no verdict). Timeout sources are excluded
from the verification rate calculation rather than penalised.
#13
HIGH
Fetching
Fuzzy Match on Full Page Text Is Semantically Meaningless
■  ISSUE
FuzzyWuzzy partial_ratio of a 10-word claim against a 5,000-word article returns high scores for any page
containing the same keywords — regardless of whether the page supports or contradicts the claim. 'Coffee
does NOT reduce cancer risk' fuzzy-matches highly with a claim that it does.
✓  FIX
Replace keyword fuzzy matching with semantic similarity using a sentence-transformer (e.g.,
all-MiniLM-L6-v2). Embed the agent's claim and each page's key sentences. A cosine similarity >0.75 on
relevant sentences = VERIFIED. This catches semantic support, not keyword coincidence.
★  NOVELTY EDGE
Semantic sentence-level verification using embeddings is more accurate and computationally efficient than
full-document fuzzy matching. It also enables verification of paraphrased claims — a known gap in
keyword-based systems.

#14
HIGH
Fetching
No User-Agent Header — Instant Bot Detection
■  ISSUE
requests.get() sends 'python-requests/2.x.x' as the user-agent. Every major news, medical, and scientific
site blocks this immediately. Fetch failures caused by bot detection are indistinguishable from URL
non-existence in the current implementation.
✓  FIX
Set a realistic browser user-agent string on all requests. Add rotating User-Agent headers from a pool of
real browser strings. Add Referer headers. For sites requiring JavaScript, use httpx with http2 support.
These are standard scraping hygiene practices.
#15
MEDIUM
Fetching
No Paywall / Login-Wall Detection
■  ISSUE
Paywalled articles (Nature, Lancet, NEJM) return HTTP 200 with login-wall HTML, not the actual article.
BeautifulSoup extracts the login page text, which matches almost nothing, incorrectly labelling the source
as UNVERIFIED.
✓  FIX
After fetching, check if the extracted text contains paywall markers ('Subscribe to read', 'Sign in to
access', word count < 200). Label such sources as PAYWALL_RESTRICTED rather than UNVERIFIED. Fall back to
PubMed abstract API for DOI-linked sources.
■ LangGraph Architecture
#16
CRITICAL
LangGraph
Graph Recompiles on Every Streamlit Rerun
■  ISSUE
Streamlit reruns the entire Python script on every user interaction. If the LangGraph StateGraph is
compiled outside of @st.cache_resource, the graph recompiles on every click — expensive and also resets
in-progress graph state mid-execution.
✓  FIX
Wrap the graph compilation in @st.cache_resource(show_spinner=False). The compiled graph is instantiated
once per app lifecycle and shared across reruns. Create a new thread_id (UUID) per user session in
st.session_state to isolate concurrent users.

#17
CRITICAL
LangGraph
Concurrent Users Share Graph State — State Bleeding
■  ISSUE
On Streamlit Cloud, multiple users share the same Python process. Without explicit thread isolation, two
users submitting claims simultaneously produce graph state objects that can interleave — verdicts mixing
evidence from two different claims.
✓  FIX
Use LangGraph's MemorySaver checkpointer with a unique config={'configurable': {'thread_id': session_uuid}}
per user. Each thread has isolated state. Add a ThreadPoolExecutor with max_workers=4 to cap concurrent
graph executions.
#18
HIGH
LangGraph
No LangGraph Checkpointing — Crashes Lose All Progress
■  ISSUE
Without checkpointing, a crash at Round 2 of 3 (Groq rate limit, timeout) silently loses the entire debate.
The user sees an error with no partial results, no recovery path.
✓  FIX
Add SqliteSaver checkpointer for local runs and PostgresSaver for production. Partial debate results are
persisted at each graph node. On failure, the UI shows completed rounds and allows resume from the last
checkpoint.
★  NOVELTY EDGE
Resumable debates are a unique UX capability — if a claim takes 90 seconds to process and fails at Round 3,
users can resume instead of restarting. No other open-source fact-checker offers this.
#19
HIGH
LangGraph
No Human-in-the-Loop Override Before Final Verdict
■  ISSUE
For a fact-checking system whose verdicts affect how people understand scientific and medical claims, there
is no human review step before the verdict is issued. The Moderator's LLM output is always accepted as
final.
✓  FIX
Use LangGraph's interrupt_before on the ModeratorNode. For high-stakes claims (medical, political, legal),
surface an 'Approve / Override / Flag for Expert Review' UI step before the verdict is stored or shown. Log
all overrides for model improvement.
★  NOVELTY EDGE
Human-in-the-loop moderation before LLM verdict finalisation is a research-grade safety feature. It creates
a feedback dataset for fine-tuning and aligns with emerging AI accountability requirements.

#20
CRITICAL
Token/Model
Context Window Overflow — Round 3 Runs on Truncated History
■  ISSUE
Each round passes the full transcript of all prior rounds. By Round 3, context = system prompt + R1_Pro +
R1_Con + R2_Pro + R2_Con + R3 instruction. This likely exceeds 4,000 tokens per call. Groq silently
truncates, meaning Round 3 agents argue from an incomplete debate history.
✓  FIX
Implement a DebateSummariser that compresses prior rounds into a 300-token summary before each new round.
Use a sliding window: agents see the summary of all previous rounds + the full last round. Track token
count explicitly and warn if approaching limits.
#21
CRITICAL
Token/Model
Structured Output Parsing From Llama 3.1 Is Unreliable
■  ISSUE
Llama 3.1 70B produces inconsistent JSON without constrained decoding. A response like 'The claim appears
mostly true, around 72%' breaks any parser expecting {verdict: 'PARTIALLY TRUE', confidence: 0.72}. Silent
parse failures mean corrupted downstream verdicts.
✓  FIX
Use Groq's JSON mode (response_format={type: 'json_object'}) with a strict Pydantic schema: class
DebateOutput(BaseModel). Add a retry-on-parse-failure loop (max 2 retries). If parsing fails after retries,
return a safe fallback verdict of INSUFFICIENT_EVIDENCE.
#22
HIGH
Token/Model
Gemini Fallback Mid-Debate Creates Inconsistent Debater Identity
■  ISSUE
If Groq fails mid-debate, Round 1 was argued by Llama 3.1 70B and Round 2 is argued by Gemini Flash — two
models with different knowledge cutoffs and reasoning styles. The weighted verdict system doesn't track
which model argued which round.
✓  FIX
Record the model_id used for each agent call in the graph state. The Moderator receives this metadata and
can weight arguments differently based on model provenance. Add a debate_integrity_flag = MIXED_MODELS when
fallback is triggered, shown to the user as a caveat.

The 23% Hallucination Rate Statistic Is Itself Unverified
■  ISSUE
The README states 'Detects hallucinations (23% of AI responses)' with no methodology, no test set, no

■ Testing Gaps
#24
HIGH
Testing
Tests Cover Plumbing — Agent Intelligence Is Untested
■  ISSUE
38 tests cover URL validation, fuzzy matching, and verdict math — all deterministic functions. The actual
LLM agent behaviour (coherent arguments? sensible verdicts? correct FactChecker interpretation?) is almost
certainly mocked. '100% passing' measures infrastructure, not product quality.
✓  FIX
Add an eval suite using LangSmith or a custom evaluator: 20 golden claims with known ground-truth verdicts.
Run weekly against real API calls (not mocks). Track verdict accuracy, confidence calibration, and
hallucination rate over time as regression metrics.
★  NOVELTY EDGE
LLM eval regression testing tied to ground-truth claim outcomes is a production ML practice rarely seen in
open-source agent projects. It creates a continuous quality signal as models are updated.
#25
HIGH
Testing
No Adversarial / Red-Team Test Cases
■  ISSUE
All example test claims are benign. There are no tests for prompt injection ('Ignore previous
instructions'), extremely long claims, compound claims, claims in other languages, or offensive claims that
should be rejected pre-debate.
✓  FIX
Add a red_team/ test directory with: 5 prompt injection variants, 3 compound claims, 2 non-English claims,
2 offensive claims that should trigger content filter, 1 extremely long claim (500+ words). Each should
have an expected_behaviour assertion.

#26

Testing
No CI/CD Pipeline — '100% Tests Passing' Is an Unverified Claim
■  ISSUE
There is no GitHub Actions workflow. The test status badge is a manually written claim in the README, not a

■ Streamlit & Operations
#27
HIGH
Streamlit
UI Freezes for 90+ Seconds — No Streaming or Async
■  ISSUE
3 rounds × 2 agents + FactChecker + Moderator, all synchronous. Streamlit's execution model means the UI is
completely frozen and unresponsive during the entire debate. Users have no feedback and will assume the app
crashed.
✓  FIX
Use Streamlit's st.status() and st.write_stream() for real-time output. Stream each agent's response as it
is generated using Groq's streaming API. Show a live progress tracker: Round 1 ✓ → Round 2 ■ → FactChecker
pending. Debate feels alive, not frozen.
★  NOVELTY EDGE
Live-streamed adversarial debate with per-round progress indicators is a distinctive UX that no existing
web-based fact-checker offers. It makes the reasoning process transparent and engaging.
#28
HIGH
Streamlit
No Verdict Caching — Same Claim Re-Runs 12+ API Calls
■  ISSUE
Submitting the same claim twice triggers a full debate re-run: 12+ LLM calls, ~90 seconds, burns free-tier
quota. There is no caching of any kind.
✓  FIX
Add semantic verdict caching: embed the input claim, check cosine similarity against cached claims in a
SQLite vector store. If similarity > 0.92, return the cached verdict instantly. Cache invalidation is
time-based (7 days). Show the user 'Cached result from [date]'.
★  NOVELTY EDGE
Semantic (not exact-match) caching of fact-check verdicts is novel — two phrasings of the same claim
('vaccines cause autism' vs 'do vaccines cause autism?') correctly hit the same cache entry.

#29
MEDIUM
Streamlit
Memory Leak in Moderator Chat on Long Sessions
■  ISSUE
The Moderator Decision Chat appends to an in-memory message list in st.session_state with no maximum size.
Long chat sessions accumulate unboundedly. On Streamlit Cloud with 1GB RAM, this can crash the app
mid-session.
✓  FIX
Cap message history at 20 messages. When the limit is reached, summarise the oldest 10 messages into a
single context_summary entry using a lightweight LLM call (Haiku 4.5). The summary replaces the raw
messages, preserving context without memory growth.
#30
MEDIUM
Product
No User Feedback Loop — Every Wrong Verdict Disappears
■  ISSUE
Users have no way to mark a verdict as wrong. Every incorrect verdict the system produces is lost — no
signal for improvement, no correction dataset, no way to detect systematic errors. For a fact-checking
system, this is a critical missing feature.
✓  FIX
Add a thumbs-up/thumbs-down feedback button under every verdict. Store feedback in a SQLite table: (claim,
verdict, user_feedback, timestamp). Add a /admin dashboard showing verdict accuracy by category. Use
disagreement signals to trigger human review and build a fine-tuning dataset.
★  NOVELTY EDGE
A public feedback-driven verdict correction loop creates a flywheel: more users → more corrections → better
model → more accurate verdicts. This is the core loop that made Wikipedia more reliable than Britannica.

Novelty Roadmap — What Makes InsightSwarm Unique
Implementing the fixes marked with ★ NOVELTY EDGE yields capabilities not present in any existing open-source
fact-checker:
★ Retrieval-Augmented Adversarial Debate (RAAD)
Agents receive Tavily-retrieved snippets before arguing. Arguments are grounded in retrieved evidence, not parametric
hallucination. Source verification becomes trivial — the system already knows what sources exist because it retrieved them.
★ Mid-Debate Verification Gates
Failed source verification in Round 1 feeds back into Round 2 as constraints. This is iterative evidence refinement — no other
open-source system does this.
★ Heterogeneous-Model Adversarial Pairing
Llama 3.1 70B vs Gemini 2.0 Flash produces genuinely divergent arguments from different training corpora. This is
academically validated as producing higher-quality debate outcomes than single-model systems.
★ Transparent Decomposed Confidence Scores
Users see WHY the system is confident: argument quality 40% + verification rate 35% + consensus 25%. No existing
consumer fact-checker exposes this breakdown.
★ Scientific Consensus Pre-Gate
Settled science claims are routed away from adversarial debate entirely. This prevents the system from implying controversy
where scientific consensus is clear — a failure mode that has damaged public trust in AI tools.
★ Atomic Claim Decomposition
Compound claims are split into atomic sub-claims before debate. Each sub-claim gets an independent verdict. This mirrors
how professional human fact-checkers operate.
★ Semantic Verdict Caching
Near-duplicate claims hit the same cache entry using embedding similarity. Computationally efficient and user-visible — two
phrasings of the same claim correctly return the same verdict.
★ Resumable Debates via LangGraph Checkpointing
Long-running debates that fail mid-execution can be resumed from the last checkpoint. A unique UX capability with no
equivalent in existing fact-checkers.

The core innovation of InsightSwarm — hallucination detection via URL verification combined with weighted verdicts — is genuinely

before any public launch.
